# SHOWCASE block templates

Rules that apply to every variant:

- Keep the markers exactly as written — they're what makes re-runs idempotent.
- Relative asset paths (`docs/showcase/hero.webp`), never raw.githubusercontent
  URLs — they work on GitHub, in local previews, and in forks.
- Hero image links to the live site.
- Mobile / full-page shots go inside `<details>` so the README stays scannable.
- Keep a blank line between any raw `<img>`/`<p>` HTML and a following `###`
  heading — without it, CommonMark swallows the heading into the HTML block.

## Multi-page app / single-purpose tool (default)

```markdown
<!-- SHOWCASE:START — generated by readme-showcase; edit docs/showcase/showcase.scenario.json and re-run instead of hand-editing -->
## ✨ See it in action

<a href="https://the-live-url.example.com">
  <img src="docs/showcase/hero.webp" alt="Project name — home page" width="100%">
</a>

![Demo: typing custom text and switching templates](docs/showcase/demo.gif)

<details>
<summary>📱 More screenshots</summary>

| Mobile | Templates gallery |
|---|---|
| <img src="docs/showcase/mobile-home.webp" width="260"> | <img src="docs/showcase/templates-full.webp" width="520"> |

</details>

<!-- docs/showcase/demo.mp4 holds a higher-quality version of the demo — drag it into a GitHub release or comment if you want crisp playback. -->
<!-- SHOWCASE:END -->
```

Adapt the `<details>` table to however many secondary shots exist (one column
per shot, mobile shots ~260px wide, desktop ~520px).

## Placement

- Markers already present → replace only the content between them.
- Otherwise insert **after** the title/badges/intro paragraph and **before**
  the table of contents or first `##` section.
- If the README already has a hand-made screenshots section, ask the user
  whether to replace it or keep both — don't silently duplicate.
